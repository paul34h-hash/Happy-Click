# 배송 관리 — 정기 배송 기사용 로컬 저장 웹앱

외부 서버·데이터베이스 없이, 브라우저 `LocalStorage`만으로 동작하는 단일 `index.html` PWA.
고정 배송 구역(아파트/동)을 한 번 등록해두고, 날짜별로 목표 수량 설정 → 실시간 배송 기록 →
정형화된 리포트 생성·복사까지 한 기기 안에서 처리한다.

**배포 주소**: https://paul34h-hash.github.io/Happy-Click/ (GitHub Pages, `main` push 시 자동 배포)

## 화면 구성 (하단 탭 5개)
1. **배송지 등록/관리** — 아파트/지역명(필수) + 동(선택)을 등록·수정·삭제, 아파트별 그룹핑,
   ▲▼로 동 순서·아파트 순서 재배열(= 실제 배송 동선). [목록 공유]/[목록 가져오기]로
   다른 기기(사장님↔기사님 폰)에 배송지 마스터를 링크 하나로 전달 가능
2. **배송 준비** — 등록된 배송지별로 금일 목표 수량 입력 (선택 사항, 비워도 진행 가능),
   아파트별 아코디언 + 입력 후 Enter로 다음 동으로 자동 이동
3. **배송중** — 배송지 카드 목록(목표 대비 진행률 배지) → 카드 선택 시 호수 직접 입력(숫자패드)
   → 수량 조정 후 추가. 상세 상단 배지는 목표가 있으면 "배송/준비개", 없으면 "배송개".
   [다음 동으로] 버튼으로 목록 복귀 없이 바로 다음 배송지로 이동. 오타 시 개별 항목 삭제 가능
4. **완료** — 금일 리포트를 정형 텍스트로 생성 → [텍스트 복사하기] / 공유(Web Share API)
5. **히스토리** — 로그가 있는 과거 날짜를 리스트로 조회, 날짜 선택 시 해당일 리포트 재확인·복사·삭제,
   상단에서 기준일 이전 기록 일괄 삭제 (용량 관리용)

상단 헤더의 날짜 선택기는 탭 2~4(준비/배송중/완료)가 공통으로 참조하는 "작업 날짜"이며,
히스토리 탭은 이 날짜와 무관하게 로그가 존재하는 모든 날짜를 독립적으로 조회한다.

> 2026-07-04: 층수+라인 자동계산("빠른 입력") 기능은 현장에서 쓰지 않는다는 피드백으로 제거.
> 호수는 항상 직접 입력한다.

## 리포트 출력 포맷
```
[2026-07-02 배송 보고]
■ A아파트 101동: 총 10개 (상세: 101호 2개, 401호 1개, 502호 7개)
■ A아파트 102동: 총 5개 (상세: 203호 3개, 105호 2개)
```
동이 없는 배송지는 `■ 아파트명: 총 N개 (...)`처럼 동 이름 없이 표시된다.

## 설계상 주요 결정

| 항목 | 결정 | 이유 |
|---|---|---|
| 데이터 저장 | `LocalStorage` 단일 키(`deliverylog_v1`)에 JSON으로 저장 | 배송 기사 본인 폰에서만 조회하면 되는 단일 사용자 워크플로우라 서버 동기화가 불필요함 |
| 데이터 모델 | `locations`(날짜 무관 마스터) + `plans[date]`(목표) + `logs[date]`(기록, apt/dong 이름을 기록 시점에 스냅샷) | 배송지를 나중에 수정·삭제해도 과거 히스토리 리포트가 깨지지 않도록 이름을 로그에 함께 저장 |
| 배송지 순서 | 번호 자동정렬이 아니라 등록/재배열(▲▼) 순서를 그대로 유지 | 실제 배송 동선은 건물 번호 오름차순과 다를 수 있음 (기사 피드백) |
| 오타 대응 | 배송중 상세 화면에서 개별 기록 항목 삭제 가능 | 등록만 되고 수정 수단이 없으면 실사용에서 불편함 |
| 마지막 상태 복원 | 선택 날짜·활성 탭·열려 있던 배송지 상세를 `LocalStorage`에 저장 | 새로고침/앱 전환 후에도 하던 작업을 이어서 진행 |
| 오프라인 대응 | Service Worker: HTML은 Network First, 정적 자산은 Cache First | HappyClick(단지패스) 프로젝트 학습 반영 — 캐시 고착으로 인한 업데이트 누락 방지 + 오프라인 재실행 속도 보장 |
| 배포 파이프라인 | GitHub Actions(`actions/deploy-pages`) | "브랜치에서 배포" 방식이 push 후 빌드를 생성하지 않는 누락이 2026-07-03 하루 3회 발생해 교체 (아래 트러블슈팅 참고) |
| 관리 화면 보호 | 없음 (비밀번호 게이트 미적용) | 배송 기사 본인 기기에서만 쓰는 개인용 앱이라 별도 잠금이 불필요하다고 판단 |
| 카카오톡 공유 | `navigator.share()` (Web Share API), 미지원 시 클립보드 복사 폴백 | Kakao SDK 앱키 없이 바로 사용 가능 |

## 트러블슈팅 기록 (다음에 비슷한 문제가 생기면 여기부터 볼 것)

### GitHub Pages 배포가 push해도 반영 안 됨 (2026-07-03)
"브랜치에서 배포(Deploy from a branch)" 방식이 간헐적으로 **빌드 자체를 생성하지 않고 조용히 실패**했다
(하루 3회 확인, 실패해도 알림이 잘 안 보임). `.github/workflows/deploy.yml`에서
`actions/upload-pages-artifact` + `actions/deploy-pages`를 쓰는 방식으로 교체 후 해결.
- **배포 확인은 반드시 이 순서로**: ① `gh api repos/paul34h-hash/Happy-Click/actions/runs`로
  "Deploy to GitHub Pages" 워크플로의 `conclusion: success` 확인 → ② `curl`로 실제 배포된
  `index.html` 원문에서 방금 넣은 코드가 있는지 grep. **WebFetch로 라이브 사이트를 확인하지 말 것**
  — 마크다운 변환 과정에서 `<script>`/`<style>` 내용이 통째로 제거돼 "반영 안 됨"으로 오판하기 쉽다
  (실제로 한 번 오판한 적 있음).
- 그래도 워크플로가 안 도는 경우 `git commit --allow-empty`로 재트리거.

### 아이폰에서만 하단 탭바 아래 빈 공간이 생김 (2026-07-04, 최종 해결)
안드로이드는 정상인데 iOS PWA(홈 화면에 추가한 앱)에서만 하단에 빈 띠가 남는 문제.
시행착오 순서와 각 원인은 아래와 같다 — **비슷한 레이아웃 버그가 다시 나오면 헤더의 달력 아이콘을
5번 연속 탭**하면 `innerHeight`/`screen.height`/`visualViewport` 값을 보여주는 진단 팝업이 뜨니
추측 대신 이 수치로 원인을 좁힐 것.

1. (오진) `#app`을 `position: fixed`로 전체화면 고정 + `overscroll-behavior: contain` — iOS 스크롤
   바운스가 하단 네비를 들어올리는 일반적인 패턴이라 우선 적용했으나 근본 원인은 아니었음.
2. (오진) 입력창 폰트 16px 미만이면 iOS가 포커스 시 확대(줌)하고 안 풀리는 현상 → 모든 input을
   16px로 강제. 유효한 별개 버그였지만 이번 증상의 주 원인은 아니었음.
3. (부분 원인) iOS 26 WebKit 버그([#297779](https://bugs.webkit.org/show_bug.cgi?id=297779)) —
   키보드가 닫힌 뒤 `visualViewport.offsetTop`이 0으로 복귀하지 않아 고정 레이어가 밀린 채 남는
   현상. `offsetTop` 실측 후 `translateY`로 역보정하는 코드를 추가(오작동 시 자동 해제).
4. **(확정 원인)** 진단 팝업 실측 결과 `innerHeight: 793 / screen: 852` (차이 59pt), `offsetTop: 0`,
   `scale: 1` — 밀림도 줌도 아니었고, **iOS 26이 `apple-mobile-web-app-status-bar-style:
   black-translucent`(투명 상태바) 모드의 홈 화면 앱에서 웹뷰 높이 자체를 상태바 높이(59pt)만큼
   짧게 계산**하는 것이 원인이었다. 웹뷰 바깥 영역이라 JS/CSS로는 보정 불가능. `index.html`의
   해당 meta를 `content="default"`로 변경해 해결.
   - **주의**: 이 meta는 iOS가 "홈 화면에 추가"하는 시점에 고정된다. 이미 설치된 기존 PWA에는
     캐시 초기화만으로 적용되지 않으며, **앱 삭제 후 재설치**해야 반영된다 (LocalStorage도 함께
     삭제되므로, 삭제 전 [목록 공유]로 배송지 백업 권장).
   - 참고: 재설치 후 하단에 남는 약간의 여백은 버그가 아니라 iOS 홈바(Home indicator) 예약
     영역으로, 모든 iOS 앱에 공통으로 적용되는 정상 동작이다.
